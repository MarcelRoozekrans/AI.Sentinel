# Dashboard 2.0 — Design

**Status:** approved 2026-05-02
**Target version:** AI.Sentinel 1.8.0 (minor bump — six additive `feat:` items)
**Source repo:** `C:\Projects\Prive\AI.Sentinel\` (currently 1.7.1)
**Backlog items addressed:** Dashboard category chips, Live audit log search, Per-session timeline view, Severity trend chart, Dark mode, Export audit log (NDJSON).

The three other Dashboard backlog items (threat heatmap, detector correlation matrix, alert acknowledgment) are explicitly deferred — see [`docs/BACKLOG.md`](../BACKLOG.md) where each carries an inline `_Deferred from Dashboard 2.0 (May 2026)..._` annotation with rationale (commit `a655e07`).

---

## Problem

The dashboard today is a useful surface — a TRS gauge, a stats panel, a live audit-event feed, a pending-approvals panel, and a single `Authorization` filter chip. But it lacks the operator-grade workflow primitives that turn "useful read-only feed" into "incident triage tool":

1. **No category filtering beyond AUTHZ.** The chips bar implies the rest (`Security`, `Hallucination`, `Operational`) but those chips were never wired. Today operators eyeball the whole feed.
2. **No free-text search.** When an operator says *"find every event mentioning `delete_database`"*, they have to grep the raw audit log.
3. **No per-session drill-down.** The session-id column is informational only; operators investigating an incident in session `abc123` have no in-dashboard way to filter to that session.
4. **No at-a-glance trend.** The TRS gauge shows the current value; nothing shows where the score is *going*. Operators can't see "spiked 4 minutes ago" without leaving the page.
5. **Dark-mode-only.** Operators on light-mode operating systems get a dark dashboard regardless. Minor, but unprofessional.
6. **No export.** Compliance asks ("send me the audit trail for the last hour") force the operator to dump the SQLite file or hit the audit forwarder out-of-band.

Each of the six is small in isolation; bundled, they tell a unified story for a release entry: *"Dashboard 2.0 — operator-grade incident triage."*

## Goals

1. Ship six dashboard items in a single coherent cycle.
2. Preserve the existing tech disposition: HTMX + inline SVG + raw CSS, **no new JS framework**, **no new external dependencies**.
3. Every interactive feature flows through URL query parameters that intersect (chips ∧ search ∧ session). The URL captures the full operator state — deep-linkable, shareable.
4. Strict additivity. Existing `/api/feed` URLs without query params return the same global feed they did before.
5. Dark mode is auto-only via `prefers-color-scheme` — zero JS, zero state, zero new endpoint.

## Non-goals

- Threat heatmap, detector correlation matrix, alert acknowledgment, detector hit-rate sparklines (per-detector trends), multi-instance aggregation — all deferred per BACKLOG annotations.
- Adding a JS framework (React/Vue/Alpine etc.).
- Refactoring the existing TRS gauge or approvals panel.
- Manual dark-mode toggle persisted in localStorage (auto-only for now; manual toggle is a follow-up only if operators ask).
- Server-sent-events for the trend chart (HTMX poll is sufficient).

## Design decisions

### D1. Category-chip classification logic — DetectorId prefix match

`?filter=security` → `e.DetectorId.StartsWith("SEC-", Ordinal)`. Same shape for `hallucination` (`HAL-`), `operational` (`OPS-`), `authorization` (`AUTHZ-`). Internal helper `IsInCategory(string detectorId, string category)` collapses the four into one switch.

**Rejected alternatives:**
- New `DetectorCategory` enum on `IDetector` — would force a public-API change for every existing third-party detector.
- Hybrid (prefix-match for built-ins, attribute for opt-in) — two sources of truth + reflection scan. Defer until a real third-party with non-conforming IDs surfaces.

### D2. Per-session view — filter on existing feed (no new route)

URL param `?session=<id>`. Same filter pipeline. Composes cleanly with chips + search (all three intersect). Operator clicks a session-id link in the table → URL gains `?session=<id>` → feed body re-renders.

**Rejected alternatives:**
- Separate `/sessions/{id}` route — doubles template surface, severs filter context (operator was investigating session X *because* a SEV-HIGH event fired there).
- Side-panel slide-out — JS-heavy interaction inconsistent with the rest of the HTMX-only dashboard.

### D3. Search — server-side, free-text against `Summary`

URL param `?q=<text>`. SQLite store: `WHERE summary LIKE '%' || $q || '%' COLLATE NOCASE`. Ring-buffer store: in-memory `Contains(StringComparison.OrdinalIgnoreCase)`. UI: search input box in the chips bar with `hx-trigger="keyup changed delay:300ms"` so typing throttles the request.

**Rejected alternatives:**
- Client-side filter — would require shipping JS; breaks the existing pattern.
- FTS5 (SQLite full-text search index) — overkill for the audit table's volume; pre-1.0 feature.

### D4. Severity trend chart — 15-min rolling, 30-second buckets, inline SVG line

New endpoint `GET /api/trend` returns `[{ t: epochMs, count: int, maxSev: int }, ...]` — 30 buckets × 30 seconds = 15-minute rolling window. Render: inline SVG line chart, ~120px tall, x-axis = time, y-axis = count (linear). Stroke color modulates by `maxSev` (green→amber→orange→red). HTMX poll every 5s.

**Rejected alternatives:**
- Client-side chart library (Chart.js, etc.) — new dependency, breaks the no-JS-framework discipline.
- Server-sent events — HTMX poll handles 5-second cadence fine.
- Operator-tunable window — adds UI surface area; 15-min is a sensible default for incident triage.

### D5. Dark mode — auto-only via `@media (prefers-color-scheme: light)`

`wwwroot/sentinel.css` gains `@media (prefers-color-scheme: light) { :root { --bg: ...; --surface: ...; ... } }` overrides. Pure CSS, zero JS, zero state.

**Rejected alternatives:**
- Manual toggle persisted in localStorage — adds JS + state management; no operator has asked.
- Theme-name attribute on `<html>` driven by JS — same overhead.

### D6. NDJSON export — current filtered view, button in chips bar

New endpoint `GET /api/export.ndjson?<query string>` — same query string as the feed, returns matching entries as NDJSON. `Content-Type: application/x-ndjson; charset=utf-8`, `Content-Disposition: attachment; filename="audit-YYYYMMDD-HHmmss.ndjson"`. Reuses `AuditJsonContext.Default.AuditEntry` source-gen serializer.

UI: small "📥 Export" button in the chips bar (right-aligned, after the search input), normal `<a href>` for the download — no JS needed.

**Rejected alternatives:**
- Always-everything export (ignores filters) — operator gets too much; if they want all entries, they clear filters first.
- Multiple format options (CSV, JSON-array, NDJSON) — NDJSON matches every existing audit forwarder and SIEM; one format keeps the UI simple.

## Architecture

All work lives in `src/AI.Sentinel.AspNetCore/`. No new packages, no new dependencies.

### Endpoints

| Method | Path | Purpose | Notes |
|---|---|---|---|
| GET | `/api/feed?filter=&q=&session=` | Live feed (existing, extended) | New query params; existing param `filter=authz` already shipped |
| GET | `/api/trend` | Trend chart data | New; returns SVG path string OR JSON for client-side render — pick SVG for AOT-friendliness |
| GET | `/api/export.ndjson?filter=&q=&session=` | Filtered NDJSON download | New; same filter pipeline as `/api/feed` |
| GET | `/api/stats` | Per-severity counts (existing) | Unchanged |
| GET | `/api/trs` | TRS SSE (existing) | Unchanged |

### Filter pipeline (single source of truth)

A new internal helper `FilterAuditEntries(IEnumerable<AuditEntry> entries, string? filter, string? q, string? session)` lives in `DashboardHandlers.cs` and is called by `LiveFeedAsync`, `TrendAsync`, and `ExportNdjsonAsync`. Consistent semantics; one place to fix bugs.

```csharp
internal static IEnumerable<AuditEntry> FilterAuditEntries(
    IEnumerable<AuditEntry> entries, string? category, string? q, string? session)
{
    if (!string.IsNullOrEmpty(category))
        entries = entries.Where(e => IsInCategory(e.DetectorId, category));
    if (!string.IsNullOrEmpty(session))
        entries = entries.Where(e => string.Equals(e.Session?.Value, session, StringComparison.Ordinal));
    if (!string.IsNullOrEmpty(q))
        entries = entries.Where(e => e.Summary.Contains(q, StringComparison.OrdinalIgnoreCase));
    return entries;
}

internal static bool IsInCategory(string detectorId, string category) => category switch
{
    "security"      => detectorId.StartsWith("SEC-",   StringComparison.Ordinal),
    "hallucination" => detectorId.StartsWith("HAL-",   StringComparison.Ordinal),
    "operational"   => detectorId.StartsWith("OPS-",   StringComparison.Ordinal),
    "authorization" => detectorId.StartsWith("AUTHZ-", StringComparison.Ordinal),
    _               => true,
};
```

### Trend chart rendering

`/api/trend` returns a small JSON envelope OR an HTMX-targetable HTML fragment with the SVG inline. **Pick HTMX-fragment** so the response can be `hx-swap="innerHTML"`'d into a target div — keeps the whole pipeline JS-free.

Bucket logic:
```csharp
var now = DateTimeOffset.UtcNow;
var windowStart = now.AddMinutes(-15);
var bucketMs = 30_000L;
var entries = audit.Query(...).Where(e => e.Timestamp >= windowStart);
var buckets = new (long t, int count, int maxSev)[30];
foreach (var e in entries)
{
    var idx = (int)((e.Timestamp - windowStart).TotalMilliseconds / bucketMs);
    if (idx < 0 || idx >= 30) continue;
    buckets[idx].count++;
    buckets[idx].maxSev = Math.Max(buckets[idx].maxSev, (int)e.Severity);
}
// Build SVG path d="M0,Y0 L4,Y1 L8,Y2 ..." with stroke color from maxSev
```

### Wire format

- `AuditEntry` NDJSON now includes a `"SessionId": <string|null>` field on every entry. Pre-1.8.0 entries (legacy rows in stores migrated v2 → v3) serialize as `null`. Forward-compatible: consumers that ignore unknown fields are unaffected.
- Custom response header `X-Sentinel-Truncated: true` signals that the store-level cap was hit and older matching entries were not returned. Header is omitted (not set to "false") when truncation did not occur. Convention for future Sentinel-specific headers: `X-Sentinel-*` namespace prefix; literal string values.
